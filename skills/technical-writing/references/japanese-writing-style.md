# Japanese Writing Style

Use this reference during translation, Japanese editorial review, and final
publication QA. The source image was used only for manual transcription and is
not packaged with the skill.

This reference also adapts task-relevant ideas from:

- <https://gist.github.com/k16shikano/fd287c3133457c4fd8f5601d34aa817d>
- <https://zenn.dev/ikora/articles/88f62861a953c1>

## 1. 日本語の文体

- 翻訳調を避ける。英文を直訳したような構文や語順にしない。
- 冗長表現を削る。
  - NG: 「〜することができます」
  - OK: 「〜できる」
  - NG: 「〜する必要があります」
  - OK: 「〜が必要」「〜する」
  - NG: 「〜という点において」
  - OK: 「〜は」「〜で」
- 主語を立てすぎない。
  - NG: 「私は〜と考えます」「あなたは〜してください」
  - OK: 「〜と考える」「〜する」
- 受動態を避け、能動態で書く。
  - NG: 「〜が行われる」「〜がなされる」
  - OK: 「〜する」
- 不要なカタカナ語を使わない。対象は定着していない訳語に限る。
  - NG: 「コンファーム」「インプリメント」「リクワイアメント」
  - OK: 「確認」「実装」「要件」
  - 業界で定着した語はそのまま使う。例はプルリクエスト、デプロイ、リファクタ。
- 直訳調の接続詞を避ける。
  - NG: 「加えて」「さらに重要なことに」「しかしながら」
  - OK: 「また」「重要なのは」「ただし」「だが」
- 「〜について」「〜に関して」の連発を避け、助詞でつなぐ。
- 体言止めと短文を活用し、一文を短く保つ。
- 英語構文の直訳を避ける。
  - NG: 「重要なことは X です」
  - OK: 「X が重要」
- 技術的な説明、手順、進捗メモでは、情報を圧縮しすぎず、読み手が一読で追えるように書く。
  - 矢印で関係を示さない。「A を B に移す」のように動詞で書く。
  - 1文に複数の事実を「、」や記号で詰め込まない。長くなるなら文を分ける。
  - 英単語は定着した訳語があれば訳す。
    consumer は「利用側」、retire は「廃止」、fetch は「取得」。
    write は「書き込み」、read-only は「読み取り専用」。
  - 訳すと不自然な語はそのまま使う。例は PR、commit、transaction。
  - スラッシュ略記は使わず、語を展開して書く。例は `write/tx`
    のような表記を避ける。
  - 関数名、型名、テーブル名などの識別子はバッククォートで囲み、地の文と分離する。

## 2. 技術文書の論理

日本語の技術文書では、段落を論証の単位として扱う。一つの段落には一つの話題を置き、
段落の最初の文で何を扱うかを示す。調査、判断、検証、評価が一つの段落に混ざるなら、
段落を分ける。

主張は、例や根拠が支える範囲に合わせる。例が一部しか支えないなら、主張もそこまで
狭める。未確認の推量、読者の疑念、ログからの推定、反実仮想は、断定に変えずに不確実性を
残す。

因果を述べるときは、なぜそうなるのかを一文で示す。複数の原因がある場合は一つの原因へ
まとめず、どの原因がどの結果に対応するかを分けて書く。節の中心になる用語は、使う前に
対象範囲を決める。

読み手の負荷を増やさない。後で参照しないファイル名、関数名、固有名詞は出しすぎない。
必要な具体は残し、議論に関係しない装飾的な精度だけを削る。

## 3. AI っぽさの点検

表面の記号を直す前に、書き手の立場を点検する。何を見て、何に引っかかり、どこまで
言い切るのかが本文に出ていないなら、句読点や言い換えだけを直しても不自然さは残る。

一般論に逃げない。「多くの人」「現代では」のような大きい主語で始める前に、具体的な
失敗、制約、数値、リポジトリ上の事実を出せるか確認する。出せない場合は、主張の範囲を
狭めるか、その文を削る。

モノに人間の動作をさせていないかを見る。誰が判断したか、どのログから推定したか、どの
仕組みで起きたかを名前で書く。対象を「AI」「ツール」のような広い語でぼかさず、
必要なら製品名、処理、利用者の役割まで落とす。

見出しは、節の対象か読者の問いを指す句にする。すべての見出しを結論の言い切りにすると、
各節が小さな演説のように見える。見出しで効果を狙うより、本文の順序で論理を立てる。

空虚な予告、総括、過剰な抽象語を削る。「重要」「本質」「観点」のような語で強調する
だけなら、何がどう重要なのかを具体化する。リズムは音読で確認し、同じ長さと同じ温度の
段落が続く場合は、内容に合わせて短くするか分ける。

記号の修正は最後に行う。不要な全角ダッシュ、装飾記号、感情や形容詞を囲む引用符、
中黒による長い並列を減らす。ただし、技術用語、固有名詞、コード、引用、正式な表記は
勝手に置き換えない。

## 4. 画像から文字起こしした文体メモ

日本語で書くときは、考える段階から日本語に寄せる。英語で考えてから訳すと、
横文字が多くなったり、訳文調になったりしやすい。たとえば、日本語の単語を
使っていても、言い回しだけが英語に引っ張られることがある。ただし、その分野で
普通に通じる言葉まで無理に言い換える必要はない。

強調は最小限にする。太字は、用語の初出定義や注記ラベルくらいにとどめ、
本文中の単語を強調しすぎない。

箇条書きが続きすぎると、機械的な文章に見えやすい。散文でつなげられるところは、
自然な文章としてつなげる。

専門用語は、その場で短く説明しながら使う。読み手が当然知っているものとして、
いきなり置かない。

ファイル名、パス、コード上の名前を本文にそのまま出しすぎない。共有資料や
スライドになったときに伝わりにくいことがある。必要に応じて、文書タイトルや
概念名に置き換える。

囲み枠や注記ブロックは多用しない。自然な本文に溶かせる内容なら、散文の中で扱う。
