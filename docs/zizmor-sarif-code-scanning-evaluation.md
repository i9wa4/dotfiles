# zizmor SARIF Code Scanning Evaluation

Issue #209 evaluates whether this repository should upload `zizmor` findings
as SARIF to GitHub Code Scanning.

Primary references:

- Classmethod article:
  <https://dev.classmethod.jp/articles/shuntaka-zizmor-sarif-code-scanning/>
- zizmor integrations:
  <https://docs.zizmor.sh/integrations/>
- GitHub SARIF upload docs:
  <https://docs.github.com/en/code-security/how-tos/find-and-fix-code-vulnerabilities/integrate-with-existing-tools/upload-sarif-file>

## 1. Current Repo Support

This repo already runs `zizmor` locally through pre-commit, not as a separate
Code Scanning upload.

Current surfaces:

| Surface                                  | Current state                                                                                                                                     |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| `.github/workflows/ci.yaml`              | Runs on `workflow_dispatch`, `push` to `main`, and PRs to `main`; top-level `permissions: {}`; job grants only `contents: read`                   |
| `.github/workflows/ci.yaml` checkout     | Uses SHA-pinned `actions/checkout` with `persist-credentials: false`                                                                              |
| `.github/workflows/ci.yaml` checks       | Runs `nix flake check`, `gitleaks detect`, and `betterleaks git`                                                                                  |
| `.github/workflows/release.yaml`         | Top-level `permissions: {}`; tag-only release job grants `contents: write`; actions are SHA-pinned and checkout uses `persist-credentials: false` |
| `nix/flake-parts/modules/pre-commit.nix` | Enables `actionlint`, `ghalint`, `pinact`, `zizmor`, and `actrun-lint` for workflow files                                                         |
| CI SARIF upload                          | Not configured                                                                                                                                    |

The existing setup is already close to the hardened baseline recommended by
the source article and upstream docs: default token permissions are zeroed,
job permissions are narrow, checkout credentials are not persisted, and
workflow actions are pinned.

## 2. SARIF Upload Fit

GitHub Code Scanning can display SARIF generated by third-party tools.
`zizmor` can produce SARIF with `--format=sarif`, and upstream documents two
GitHub Actions integration paths:

- `zizmorcore/zizmor-action`, which handles SARIF upload internally.
- Manual integration, where CI runs `zizmor --format=sarif` and uploads the
  result with `github/codeql-action/upload-sarif`.

For this repo, the manual path is a better prototype because it keeps the Nix
tooling path visible and avoids adding another action wrapper before alert
noise is understood.

The minimal workflow permissions for a SARIF upload job would be:

```yaml
permissions: {}
jobs:
  zizmor-sarif:
    permissions:
      contents: read
      security-events: write
```

For private repositories, GitHub and zizmor docs also mention `actions: read`
for upload/run metadata. This repository is public, so the first prototype can
try without `actions: read` and add it only if GitHub reports that it is needed.

## 3. Prototype Path

Do not make this blocking on the first branch. Run it as a separate
non-required workflow until alert volume and severity mapping are understood.

Recommended first workflow shape:

```yaml
name: zizmor-sarif

on:
  workflow_dispatch:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

permissions: {}

defaults:
  run:
    shell: bash

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  zizmor-sarif:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    permissions:
      contents: read
      security-events: write
    steps:
      - name: Checkout
        uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2
        with:
          persist-credentials: false

      - name: Install Nix
        # v31.10.6
        uses: cachix/install-nix-action@8aa03977d8d733052d78f4e008a241fd1dbf36b3

      - name: Generate zizmor SARIF
        run: nix develop .#ci --command zizmor --format=sarif . > zizmor.sarif

      - name: Upload zizmor SARIF
        # v4.36.0
        uses: github/codeql-action/upload-sarif@7211b7c8077ea37d8641b6271f6a365a22a5fbfa
        with:
          sarif_file: zizmor.sarif
          category: zizmor
```

Notes for the implementation PR:

- Keep `permissions: {}` at the workflow root.
- Keep checkout `persist-credentials: false`.
- Keep actions SHA-pinned and run `pinact`.
- Prefer the existing Nix-provided `zizmor` over installing from PyPI unless CI
  duration becomes a practical blocker.
- Preserve the current pre-commit `zizmor` hook; Code Scanning is additive.

## 4. Other CI Tool Compatibility

Current local/CI tools do not all have equal Code Scanning fit.

| Tool          | Existing use                         | Code Scanning fit                                                                                           |
| ------------- | ------------------------------------ | ----------------------------------------------------------------------------------------------------------- |
| `zizmor`      | Workflow linter in pre-commit        | Best fit; native SARIF output and GitHub Actions-specific findings                                          |
| `gitleaks`    | CI history scan and pre-commit guard | Better suited to secret scanning workflow and logs; do not duplicate secret findings into Code Scanning yet |
| `betterleaks` | CI history scan and pre-commit guard | Same as `gitleaks`; keep as secret-focused guardrail                                                        |
| `actionlint`  | Workflow syntax/static lint          | Useful local/CI check; no repo-owned SARIF upload path selected here                                        |
| `ghalint`     | Workflow security lint               | Keep local/CI for now; evaluate SARIF only if upstream support and alert quality justify it                 |
| `pinact`      | Action pinning enforcement           | Keep as deterministic pre-commit/CI enforcement, not Code Scanning                                          |

The first SARIF experiment should stay focused on `zizmor`.

## 5. Merge Blocking And Alert Visibility

`zizmor --format=sarif` is designed for upload visibility, not as a failing
exit-code gate for findings. Upstream notes that SARIF mode exits successfully
unless there is an internal audit error. Merge blocking therefore belongs in
GitHub Code Scanning alert policies or rulesets, not in the workflow step
itself.

Do not enable merge blocking until after a trial answers:

- How many alerts appear on current `main`?
- Which severities are noisy or expected for this dotfiles repo?
- Do pull request annotations identify actionable workflow lines?
- Does the public-repo permission set work without `actions: read`?
- Does the Nix-based run complete within the current CI timeout budget?

## 6. Decision

Proceed with a later focused implementation branch for a non-required
`zizmor-sarif` workflow. Keep the existing pre-commit and CI gates unchanged in
this branch.

Do not enforce merge blocking yet. First collect Code Scanning alert volume,
severity quality, and runtime cost from a manual or non-required workflow run.
